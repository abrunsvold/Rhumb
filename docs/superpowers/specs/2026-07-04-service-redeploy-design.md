# Service redeploy: blue-green replace + health gate + deploy provenance

**Date:** 2026-07-04 · **Status:** approved design
**Fixes:** F11 (BLOCKER) and F12 (HIGH) from [docs/dogfood/2026-07-04-day2-filament.md](../../dogfood/2026-07-04-day2-filament.md)

## Problem

The day-2 dogfood proved Rhumb cannot change what it built. `spawn_service` on an already-deployed id half-completed a replace: it spawned a second container, skipped the dependency install (vendored check tests `node_modules` *existence*, not completeness — `deployer.ts:23-24`), never verified the service came up (`systemctl enable --now` is the deploy's last step — `deployer.ts:55`), silently dropped the registry update (`appendService` dedupes by id — `registry.ts:21`), never rolled back (rollback only fires on a thrown exception — `ops.ts:107-113`), and left the failed container crash-looping forever (`Restart=always`/`RestartSec=2`, no start limit). The build agent then reported success because registry "health" pointed at the old container — and `status: "healthy"` is written unconditionally at deploy time (`ops.ts:103`); no health probe exists anywhere in the platform.

## Decisions (made during brainstorming)

- **Redeploy model: blue-green replace.** New container, health-gated, then atomic registry cutover, then old container destroyed. Failure before cutover = destroy new, old untouched — rollback is free and total.
- **Health gate: tiered.** HTTP if the manifest declares `healthPath`, else TCP accept on the port; both tiers also require the systemd unit active with stable restart count.
- **`spawn_service` on an existing id: hard error** pointing at `redeploy_service`. No magic.

## Design

### New gated tool: `redeploy_service { id }`

Joins the existing infra MCP server and the same `canUseTool` → pending-queue → operator-approval flow. The pending-action description states explicitly: "replaces container `<n>` for service `<id>`; the old container is destroyed after a health-gated cutover."

Flow (in `ops.redeploy(id)`):

1. **Validate:** `assertServiceId(id)`; read manifest; `manifest.id === id`; `buildExtraEnv` (fail fast on unknown data source); registry entry MUST exist — else `Error("service \"<id>\" is not deployed; use spawn_service")`.
2. **Spawn new:** same machinery as `spawn` (create → start → wait-for-IP → `deployer.deploy`), with a fresh **deployId** threaded through (format `<ISO8601 basic timestamp>-<6 hex chars>`, generated by an injected `newDeployId()` dep so tests are deterministic).
3. **Health-gate** the new container (see below). On failure: destroy the new container via the existing stop → waitStopped → destroy rollback; registry untouched; old container untouched; throw an error carrying the gate's evidence (final unit state, NRestarts, tier probed, last journal lines if retrievable).
4. **Cutover:** `replaceService(servicesPath, entry)` — atomic single-file write swapping the entry by id with new `containerId`, `host`, `deployId`, `status: "healthy"`, `updatedAt`. (Dashboard-host reads `services.json` per request; cutover is immediate, no proxy restart.)
5. **Destroy old** container (stop → waitStopped → destroy). Failure here does NOT roll back (cutover already happened): the result carries a `warning` naming the orphaned container id so it is never silent — the operator sees it in the tool result and can clean up.

Every failure path ends in exactly one of three end-states:
(a) nothing changed (pre-container validation errors);
(b) new container destroyed, old untouched, registry untouched (deploy/gate failures);
(c) cutover complete, old-container-destroy failure surfaced as a warning naming the container id.
No path leaves an unregistered running container silently.

### Health gate (`agent-host/src/services/health.ts`, new module)

`waitHealthy(target: { host, port, healthPath? , containerUnit }, deps, opts) → Promise<GateResult>`

- **Deadline:** default 90 s, overridable via `ServiceConfig.healthGateMs` (env `RHUMB_HEALTH_GATE_MS`).
- **Probe loop** (every ~5 s, injected `sleep` for tests). A pass requires, on two consecutive probes ≥ 5 s apart:
  - `systemctl is-active rhumb-<id>.service` = `active` (via the existing SshExec into the container), AND
  - `NRestarts` unchanged between the two probes (via `systemctl show -p NRestarts`), AND
  - **HTTP tier** (manifest.healthPath set): `GET http://<host>:<port><healthPath>` returns 200, on both probes; **TCP tier** (no healthPath): a TCP connection to `<host>:<port>` is accepted, on both probes.
- Probes run from the agent host (it already reaches container IPs for deploy).
- `GateResult` = `{ ok: true, probes }` or `{ ok: false, reason, lastState: { active, nRestarts, tier, httpStatus? } }` — the failure evidence embedded in the thrown redeploy/spawn error.

**`spawn_service` also gates:** the same `waitHealthy` runs at the end of `spawn` before the registry write. `status: "healthy"` now means the gate passed. Gate failure in spawn takes the existing rollback path (destroy the new container, throw). This retroactively fixes the unconditional-healthy seam (ops.ts:103).

### Manifest: `healthPath?: string`

Optional; must start with `/` (validated in `manifest.ts` beside the existing checks). Absent = TCP tier. The printer-poller's manifest gains `"healthPath": "/health"` when it is next touched (capstone, below).

### Registry (`registry.ts`)

- `appendService` — silent dedupe becomes an error: `Error("service \"<id>\" already registered")`. (Callers: only ops.spawn — which now pre-checks and errors earlier with the friendlier redeploy pointer; the registry throw is the backstop, not UX.)
- New `replaceService(path, entry)` — errors if no entry with `entry.id` exists; otherwise writes the list with the entry swapped in place (same order).
- `ServiceEntry` gains `deployId: string` and `updatedAt?: string`. Existing on-disk entries without `deployId` load fine (field optional on read, always written by new code).

### Deployer (`deployer.ts`)

- **Vendored completeness check:** vendored ⇔ `node_modules` exists AND every top-level `dependencies` key in `package.json` resolves to a directory `node_modules/<name>` (scoped names included). Otherwise install (`npm` apt package + `npm ci --omit=dev || npm install --omit=dev`, as today). This turns the day-2 failure (pg's transitive deps present, `pg` itself missing) into an install, not a crash loop.
- **Unit file additions:**
  - `StartLimitIntervalSec=60`, `StartLimitBurst=5` — a crash-looper enters `failed` within ~a minute instead of restarting at 2 s forever; the health gate's `is-active` check then fails fast with a clean signal.
  - `Environment=RHUMB_DEPLOY_ID=<deployId>` — services can echo it from their health payload (documented convention, never required by the platform).
- **Provenance file:** deployer writes `/opt/rhumb/<id>/.rhumb-deploy.json` = `{ "deployId": ..., "deployedAt": ... }` in the container.
- `deploy(...)` signature gains `deployId: string`.

### Provenance surfaced (F12)

`service_status` / `list_services` tool results and `services.json` now carry `deployId` + `updatedAt`. Ground-truth verification of "which code is live" becomes: registry deployId ↔ container `.rhumb-deploy.json` ↔ unit-file env — all platform-controlled, none dependent on app cooperation.

## Out of scope

- F7 (agent AskUserQuestion steering), all client findings (F8/F9/F14), migration story, on-ramp/deploy.sh (F15) — separate cycles.
- App-side health echo in the poller beyond the capstone manifest touch.
- In-place/releases redeploy variant (rejected in brainstorming: bespoke rollback, shared-container wedge risk).

## Testing

- **Unit (per seam):** registry replace/append-throw matrix; vendored-completeness matrix (no node_modules / complete / missing one dep / scoped dep); unit-file content (backoff lines, RHUMB_DEPLOY_ID, existing env preserved); gate state machine with fake exec/http/tcp probes (pass both tiers, HTTP-fail, TCP-fail, restart-storm → is-active fail, NRestarts flapping → no two stable probes, deadline expiry); ops.redeploy orchestration with fake lxc/deployer/gate covering end-states (a), (b), (c) including the old-destroy-failure warning and the not-deployed error; ops.spawn now gating + erroring on existing id.
- **Live capstone (the real acceptance test):** on the box, `redeploy_service { printer-poller }` through the full gate flow, shipping the staged filament fix (sitting in `<workspace>/services/printer-poller` since the day-2 turn) with `healthPath: "/health"` added to its manifest. Pass = registry shows new containerId + deployId; old container 105 gone; `pct list` shows exactly one poller container; container `.rhumb-deploy.json` matches registry; `/health` ticking; telemetry still climbing; a deliberate second redeploy with a sabotaged (dep-stripped, unvendored-marker) copy must FAIL the gate, destroy its container, and leave the healthy deployment untouched — the day-2 failure replayed against the fix.
