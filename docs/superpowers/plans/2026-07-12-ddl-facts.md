# DDL Facts Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Surface each provisioned database's DDL history (from `_rhumb.ddl_audit`) on its datasource node in the System map, per the approved F17 "surface now, gate later" decision.

**Architecture:** `infra/ddlFacts.ts` mirrors the node-facts pattern — a refresher queries each data source's database with admin credentials and writes `ddl-facts.json` atomically; the projector reads the file and decorates datasource nodes. Both external-facts refreshers compose into one `refreshExternal` used by the ontology router and `onMutate`.

**Tech Stack:** TypeScript/Express/pg/vitest, agent-host only (client renders props as-is). No new dependencies.

## Global Constraints

- Spec: `docs/superpowers/specs/2026-07-12-ddl-facts-design.md`.
- ESM `.js` import suffixes.
- Per-source degradation: one bad source never poisons the file; refresh failure never breaks `GET /ontology` or an infra op.
- DDL refresher gated on `RHUMB_PG_ADMIN` alone; node-facts stays gated on Proxmox alone.
